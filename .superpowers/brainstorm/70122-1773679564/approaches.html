Migration & Compatibility: Architecture Approaches

Three approaches for structuring the v1 to v2 migration system

A

Native Migration + React Native Compat UI

Recommended

Migration layer: Swift code in ios/Shared/ reads Core Data + old UserDefaults on first launch, converts to v2 profiles, writes via KeyboardPreferences. Runs before React Native loads.

Compat UI: New React Native screen (ClassicEditorScreen) with the flat v1-style settings list. Reads/writes v2 profile model. Available via toggle in the main editor.

Detection: Swift checks for Core Data store existence + migration flag in UserDefaults.

Pros

  • Migration happens at native layer - fast, reliable, before RN loads
  • Core Data reading is natural in Swift
  • Compat UI reuses existing RN components
  • Clean separation: native handles data, RN handles UI

Cons

  • Migration logic in Swift only (not Android) - but v1 was iOS-only anyway
  • Two separate codebases for migration (Swift) and compat UI (RN)
B

All React Native via Native Bridge

Everything in TypeScript, with a thin native bridge for Core Data reading

Migration layer: New native module V1MigrationModule exposes Core Data reading to RN. TypeScript code does the conversion logic and writes v2 profiles.

Compat UI: Same as Approach A - React Native screen.

Detection: RN calls native module to check for v1 data on app start.

Pros

  • All conversion logic in one language (TypeScript)
  • Easier to test conversion logic
  • Compat UI and migration share the same codebase

Cons

  • Core Data access from RN requires custom native module anyway
  • Migration happens after RN loads - slower first launch
  • Async bridge adds complexity for something that should be synchronous
  • More moving parts for a one-time operation
C

Hybrid: Native Migration, Shared Conversion Logic

Swift reads v1 data into a JSON blob, TypeScript does the conversion

Migration layer: Swift reads Core Data + UserDefaults, serializes all v1 templates into a JSON array, writes to a known key. On RN start, TypeScript reads this JSON, converts to v2 profiles, and saves them.

Compat UI: Same React Native screen.

Detection: Swift sets a flag when v1 data is found. RN checks flag and runs conversion.

Pros

  • Native code only does simple data extraction (no conversion logic)
  • All mapping/conversion logic in TypeScript - testable, shared with compat UI
  • Clean handoff between native and RN layers

Cons

  • Two-phase migration (extract then convert)
  • Slightly more complex than A for a one-time operation
